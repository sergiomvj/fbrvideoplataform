## Why This Exists

The system is close to compliance, but two runtime-truth concerns remain:

- render submission flow still keeps mutable in-memory cache alongside persistence
- review routing still keeps a parallel `_review_queue` in memory

These may no longer be the primary blockers, but they still weaken the “single source of truth” promise in the architecture and stories.

---

## Story

**As a** backend owner,  
**I want** render and review flows to rely on persistence as the operational truth,  
**so that** restart behavior and multi-step processing remain trustworthy.

---

## Acceptance Criteria

1. Render submission lifecycle treats persistence as primary truth, with memory only as cache if still needed.
2. Review queue operational truth does not depend on `_review_queue` surviving in memory.
3. If in-memory caches are kept, they are clearly secondary and reconstructed from persistence safely.
4. Stories `11.2` and `11.3` can be re-reviewed without architecture-truth objections.

## Detailed Implementation Instructions

### Step 1: Audit the render cache role

Open:

- reconcile and webhook paths must continue to work with empty cache

### Step 2: Audit the review queue cache role

Open:

- [routing/service.py](C:/Projetos/1FBRFaceVideo/backend/application/services/routing/service.py)
- [review_repository.py](C:/Projetos/1FBRFaceVideo/backend/infrastructure/db/review_repository.py)

Clarify the intended role of `_review_queue`:

- if it is unnecessary, remove it
- if it is retained as cache, reads and updates must still be persistence-safe

At minimum:

- no critical workflow step should fail after restart because `_review_queue` was lost
- action flows should be reproducible from persisted review items

### Step 3: Reduce truth duplication

Prefer this model:

- persistence = source of truth
- memory = optional performance cache

Do not leave ambiguous mixed behavior where some endpoints depend on DB and others on local dict state.
