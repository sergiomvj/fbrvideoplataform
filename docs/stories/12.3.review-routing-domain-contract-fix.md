## Why This Exists

The review-routing path still contains a domain-construction mismatch:

- `RoutingService` builds `ReviewQueueItem` using `candidate_id=...`
- `ReviewQueueItem` currently defines `asset_id`, not `candidate_id`

This is a likely runtime failure in the actual routing path and keeps Story `11.3` out of full compliance.

---

## Story

**As a** pipeline owner,  
**I want** the routing layer and review domain model to agree on the same contract,  
**so that** review queue creation works reliably in the live flow.

---

## Acceptance Criteria

1. `RoutingService` constructs `ReviewQueueItem` with fields that actually exist in the domain model.
2. The persisted review contract remains consistent from routing -> repository -> API response.
3. No live code path depends on mismatched names like `candidate_id` vs `asset_id`.
4. Story `11.3` becomes re-reviewable without this contract objection.

### Step 1: Choose the canonical field name

Open:

- [routing/service.py](C:/Projetos/1FBRFaceVideo/backend/application/services/routing/service.py)

- map candidate-origin values into `asset_id` when entering the review queue

Also confirm all subsequent logging and audit metadata remain semantically correct.

### Step 3: Reconcile semantic naming through the flow

If the value still originates from a “candidate”, make that explicit in metadata, not by breaking the domain constructor.

Recommended approach:

- persisted field: `asset_id`
- candidate origin detail: keep inside `candidate_data` or metadata
