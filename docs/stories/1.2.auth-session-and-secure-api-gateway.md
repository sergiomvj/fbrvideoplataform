## Story

**As a** system operator,  
**I want** authenticated access and a secure request gateway between the console and the backend,  
**so that** every protected operation in the system is executed with reliable user context and controlled exposure.

---

## Acceptance Criteria

1. Session management is implemented with `iron-session` using secure cookie settings and environment-driven secrets.
2. All frontend-to-backend protected requests flow through authenticated Next.js API/route handlers.
3. The backend receives authenticated user context in a controlled internal header or equivalent validated mechanism.
4. Protected backend routes reject unauthenticated calls consistently.
5. Error responses do not leak internals, secrets, stack traces or implementation details.
6. The authentication gateway is documented clearly enough to be reused by all future frontend stories.

### Completion Notes List

- iron-session v8 configured with lazy secret validation (avoids build-time crash)
- Session cookie: `synkra_session` — httpOnly, secure in production, sameSite=lax, 24h TTL
- Gateway pattern: catch-all route at `/api/[...path]` proxies to backend with `X-User-Id` header
- Auth session endpoints: `POST /api/auth/session` (login), `DELETE /api/auth/session` (logout)
- Backend dependency `CurrentUser` reads `X-User-Id` header, raises 401 if missing
- All error responses sanitized — no stack traces, no internal details
- Generic exception handler in FastAPI returns `{"error": "Internal server error"}` for unhandled errors
- Example protected route created at `GET /productions` with `CurrentUser` dependency
- Auth gateway pattern documented at `docs/auth-gateway-pattern.md`
