## Why This Exists

The provider-registry work improved startup wiring, but still contains runtime correctness problems:

- invalid attribute access using `provider_id` on domain objects that expose `id`
- direct mutation of registry internals
- inconsistent synchronization between `_providers` and `_providers_by_key`

This keeps Story `11.1` in `CONCERNS`.

---

## Story

**As a** runtime owner,  
**I want** provider registry synchronization to be correct and deterministic,  
**so that** provider persistence and runtime resolution behave safely in production.

---

## Acceptance Criteria

1. No code path references `p.provider_id` on `MediaProvider` domain objects.
2. Runtime sync between persistence and in-memory registry uses explicit registry methods, not direct writes into `_providers`.
3. `_providers` and `_providers_by_key` remain consistent after load, list, create, update, enable, disable and delete flows.
4. Active provider resolution works through the service layer without runtime attribute errors.
5. Story `11.1` becomes re-reviewable as `PASS`.

---

## Detailed Implementation Instructions

### Step 1: Fix the incorrect domain attribute

Open:

- [service.py](C:/Projetos/1FBRFaceVideo/backend/application/services/provider_registry/service.py)
- [provider.py](C:/Projetos/1FBRFaceVideo/backend/domain/media_sourcing/provider.py)

Replace any use of:

- `p.provider_id`

with the correct domain identity field:

- `p.id`

### Step 2: Stop mutating private registry storage directly

Today the code writes into:

- `self._registry._providers[...]`

This is not acceptable as the sync mechanism.

Refactor so that synchronization uses one of these approaches:

#### Preferred

Add an explicit method to `MediaProviderRegistry`, for example:

- `sync_provider(provider: MediaProvider)`

That method must:

- write to `_providers[provider.id]`
- write to `_providers_by_key[provider.provider_key]`
- replace any stale mapping safely

Then use that method from `ProviderRegistryService`.

#### Acceptable alternative

Use `register(provider)` only if it is guaranteed to overwrite existing entries safely in both maps.

### Step 3: Normalize service list flows

In `list_providers()` and `list_active_providers()`:

- repository results should be treated as authoritative when available
- if repository returns results, sync them through the explicit registry method
- do not partially hydrate only one of the registry maps

### Step 4: Re-check mutation flows

Verify these methods still behave correctly:

- `create_provider`
- `update_provider`
- `enable_provider`
- `disable_provider`
- `delete_provider`

Specifically confirm:

- registry lookup by id works
- registry lookup by key works
- deleted providers disappear from both maps
